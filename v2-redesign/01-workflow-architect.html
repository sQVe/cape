Perspective: Workflow Architect

Cape V2: Three Chains, One State Machine

A redesign grounded in 10 days of real usage logs — 37 sessions, 120+ skill invocations across work-platform and aburaya repos. Cape V1 is a bag of 22 skills. Cape V2 is a pipeline.
Thesis: The usage data reveals three hardened chains (Plan → Build → Ship) and a dormant bug track. V2 drops the menu metaphor and ships a state machine — phases with explicit entry/exit conditions, br epics as cross-session persistence, and five dead skills pruned without mercy.

Diagnosis

What V1 gets wrong

V1 presents 22 skills as a flat menu. Don-cape tries to route via a lookup table. This creates three failure modes the data confirms:

Problem 01
Dead weight
5 skills never fired in 10 days across two repos doing real work. They're not dormant — they're inert. Every skill adds routing surface and cognitive load for the LLM routing from don-cape.
Problem 02
debug-issue is orphaned
The designed chain is debug-issue → fix-bug → commit. In practice: debug-issue fired 1×, fix-bug fired 3×. Bugs get fixed directly without the investigation step. The chain precondition (br bug must exist) isn't enforced.
Problem 03
conform runs post-hoc
Conform appeared 7× but always after commit — as a check on work already done. Its value is real, but it's positioned as a standalone instead of a mandatory gate before closing a task.
V2 Architecture

The V2 state machine

Three primary tracks (Plan, Build, Ship) with explicit entry/exit, one bug track, and one quality gate. br epics are the only cross-session persistence.

V2 Components

Phase definitions and handoff contracts

Each phase has a mandatory entry condition, a single primary output, and an explicit exit that triggers the next phase. br epics carry state across /clear boundaries.

Phase Skills Entry condition Primary output Exit → Next Trigger
Plan brainstorm challenge write-plan No br epic for this work br epic with requirements, anti-patterns, first task User says "continue" → Build user
Build execute-plan expand-task tdd conform commit br epic open, ready task exists Passing tests, committed code, closed task All tasks closed → Ship auto
Ship finish-epic review pr All br tasks closed, epic open Merged PR, closed epic PR merged → Done auto
Bug debug-issue fix-bug commit Error / broken behavior reported br bug with root cause, minimal fix, regression test Committed fix → Done user
New in V2: conform is a mandatory gate inside Build, not a post-hoc check. It runs after tdd passes and before commit is allowed. Current V1 behavior — conform firing after commit — turns it into a retrospective that generates rework. Shifting it left means convention violations fail at the task boundary, not after the work is shipped.
V2 Changes

Three concrete changes from V1 to V2

Change 01 — Prune dead skills
Delete 5 skills. Zero survivors without 1 invocation.
Five skills fired 0× in 10 days of real work. They're not dormant features — they're routing noise. Every entry in don-cape's lookup table is a path the LLM can misfire into. Prune them.
V1: 22 skills, 5 unreachable in practice
V2: 17 skills, dead ones deleted outright
analyze-tests branch design-an-interface explain find-test-gaps
Change 02 — conform becomes a gate
Move conform left: after tdd, before commit.
Conform ran 7× in V1 — always post-commit, as a retrospective gate. That means convention violations are discovered after the work is done and committed, requiring a fixup commit. Shifting it into the Build loop as a mandatory step before commit is allowed makes convention enforcement proactive, not reactive.
V1: tdd → commit → (user runs conform)
V2: tdd → conform [gate] → commit
Change 03 — enforce bug track precondition
fix-bug requires a br bug. No shortcut entry.
debug-issue fired 1×. fix-bug fired 3×. The chain is being bypassed — bugs get fixed without root cause analysis. V2 makes fix-bug check for a br bug at entry. If none exists, it routes to debug-issue first. This isn't dogma — it's preventing fixes that mask symptoms.
V1: fix-bug fires directly, no br bug required
V2: fix-bug entry checks for br bug → debug-issue if missing
Cross-session state

br epics as the persistence layer

37 /clear boundaries in 10 days. That's 37 context resets. The only thing that survived every reset is br state — epic requirements, anti-patterns, task status, expanded plans. This is the architecture, not a detail.

What br stores (cross-session)

Epic requirements (immutable)
Anti-patterns with reasoning
Success criteria (checkable)
Task expanded plans (step-by-step)
Verify.json (checkpoint SHAs)
Outcome sections (per-task learnings)

What /clear correctly destroys

Conversation history
Agent dispatch results
Research findings (in-memory)
Design summary drafts
Intermediate reasoning
All "I was just thinking..." context
V2 design principle: If a skill needs something to survive /clear, it must write it to br before the checkpoint. expand-task writes the expanded plan to the task's design field for exactly this reason. V2 extends this pattern: conform results, divergence logs, and outcome notes are all written to br before the checkpoint stop — never left in conversation context.
