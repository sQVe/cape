The problem: flat chaos → tracked taxonomy

V1 — Flat, no grouping

22 names in one unordered bag. No lifecycle signal. No namespace. 5 skills (red) never triggered in 10 days.

brainstorm write-plan execute-plan finish-epic analyze-tests branch design-an-interface explain find-test-gaps debug-issue fix-bug conform challenge refactor beads commit pr review tdd task-refinement expand-task don-cape

Problems: debug-issue vs fix-bug (noun vs verb), conform vs challenge (adjective vs verb), beads (tool name). No mental grouping.

V2 — Six tracks, consistent verbs

Names follow track:verb. Every skill has a home. Router knows the track — loads only what's needed.

build:explore build:plan build:run build:close fix:diagnose fix:patch evolve:refactor evolve:challenge quality:review quality:conform quality:audit quality:gaps ship:commit ship:pr meta:route meta:track

Benefits: track prefix predicts content, verb suffix predicts action. Router loads the track manifest first, then the specific skill — cuts cold-start context by ~60%.

Progressive disclosure — killing the 1000-injection problem

Tier 1
Always
Injected at session start. Tiny. Just the track manifest — 6 names + one-line descriptions + chain order. <200 tokens.
meta:route build: manifest fix: manifest evolve: manifest quality: manifest ship: manifest
Tier 2
On-track
Router identifies the active track from user intent. Loads that track's skill definitions only. ~400 tokens per track vs ~1000 for full catalog.
build:explore definition build:plan definition build:run definition build:close definition
Tier 3
On-invoke
Skill loads its own agents and internal sub-skills on demand via Skill tool. Agents are never in context unless dispatched.
codebase-investigator fact-checker code-reviewer internet-researcher build:tdd (internal) build:expand (internal)

Result: a commit-only session (the most common pattern — 29 of 97 invocations) loads meta:route + ship: manifest + ship:commit — roughly 3 definitions instead of 22. Context overhead drops ~85% for single-track sessions.

Router redesign — from omniscient to thin

Dimension V1 (don-cape) V2 (meta:route)
Size Full routing table with 18 rows, all skill descriptions 6-row track dispatch + short-circuit rules only
Skill knowledge Embeds trigger text for every skill inline Delegates to track manifest; skills define their own triggers
Chain enforcement Router knows full chain; must be kept in sync with skills Each track manifest owns its chain — router just selects track
Agent dispatch Router lists all 7 agents and which skills call them Agents invisible to router; skills dispatch independently
Maintenance burden Adding a skill = update router + skill file Adding a skill = update track manifest only
Never-fire skills Still injected every session (analyze-tests, branch, etc.) Pruned or absorbed; quality:audit promoted via post-close hook

Resolving skill vs agent conceptual overlap

V1 ambiguity: code-reviewer exists as both an agent and review exists as a user-facing skill. codebase-investigator was used 30× but is invisible to users. The distinction was implicit.

Type V2 role User-visible? Invoked by
Skill Orchestrates a user-facing workflow. Has a trigger, chain position, rigidity level, stops for user input. Yes — slash command or routing User intent → meta:route
Agent Deep, sandboxed specialist. No chain, no user stops. Returns a result to the calling skill. No — internal only Skills dispatch via Agent tool
Internal skill Sub-routine within a track. Has rigidity but no user trigger. E.g. build:expand, build:tdd. Visible in docs, not routing Parent skill calls Skill tool

Rule of thumb: if a user would ever type /cape:<name>, it's a skill. If it does deep work and hands back a result, it's an agent. If it's a reusable step within one track, it's an internal skill.

Key findings from usage data

01
Five skills are dead weight
analyze-tests, branch, design-an-interface, explain, find-test-gaps fired zero times in 10 days. Each still injected every session. Either absorb into existing skills or move to opt-in tier-3 only.
02
Naming is a cognitive tax
V1 mixes naming axes: lifecycle verbs (execute-plan), problem nouns (debug-issue), action verbs (conform), tool names (beads). A newcomer cannot predict what any skill does from its name alone.
03
Router is a single point of truth AND failure
don-cape carries the entire routing table, chain logic, and agent manifest. Adding any skill requires updating the router. V2 delegates: tracks own their manifests, skills own their triggers.
04
commit dominates (29×)
The most-used skill by 2.5×. Yet it loads the same 22-skill context as a full build session. A ship-only session should load only meta:route + ship: manifest — under 3 definitions total.
05
codebase-investigator is the real workhorse
Used 30× internally but completely invisible to users. V2 formalizes the distinction: agents are black-box specialists dispatched by skills, never user-invoked. Naming reflects this — no user-facing alias.
06
quality skills lack a home
review, conform, analyze-tests, find-test-gaps are scattered in V1 with no shared identity. Grouping them under quality: makes them discoverable as a unit and enables a "quality pass" gate after build:close.
