Automation & Power-User

Hooks enforce invariants the model cannot. Skills orchestrate judgment the harness cannot. V2 draws that line sharply — and makes br the durable state machine that survives every /clear.

37 /clear sessions observed 29 commit invocations 30 codebase-investigator dispatches 22 skills · 8 agents 7 conform runs as gate tddState tracked in hooks

The core principle: hooks are deterministic C code (the harness always fires them). Skills are probabilistic (the model interprets them). Put invariants in hooks. Put judgment in skills. Never swap them.

Hook events catalog — V2 proposals

Event Matcher Action Behavior Why hook not skill
PreToolUse Edit Write DENY Block prod edits when tdd-state ∈ {null, writing-test} Model-driven TDD bypasses consistently fail under time pressure
PreToolUse Bash git commit DENY Block commit if cape conform staged returns violations Conform as gate, not suggestion — 7 manual invocations become 0
PostToolUse Edit Write OBSERVE Detect test file writes → set tdd-state=writing-test State transition must be atomic with the edit, not rely on model instruction
PostToolUse Bash test runner OBSERVE Parse exit code → tdd-state = red (fail) or green (pass) State machine correctness; model cannot reliably parse its own test output
PostToolUse Bash git commit WARN Run conform branch; append violations as additionalContext Post-commit review loop; violations visible before next task begins
UserPromptSubmit (any) INJECT If workflowActive & tdd-state=null → prepend TDD reminder Safety net for resumed sessions after /clear; skill state lost, hook state survives
Stop (any) WARN If workflowActive → remind user to /clear before next task (HITL mode) Structural checkpoint discipline cannot rely on model remembering every time
Escape hatch: cape state set tdd-bypass suspends PreToolUse deny for the current session. Manual set/clear only — no auto-clear, no TTL. Hidden coupling between flags is worse than occasional manual bypass.

TDD state machine — hook-enforced

tdd-state prod edit test edit
null (no TDD active) DENY allow
writing-test DENY DENY
red (test failing) allow allow

⟳ Step begins — execute-plan resets tdd-state → null per-step, not per-edit

✕ Write more tests? — DENY (batching) stay atomic

37 /clear events in 10 days — every session boundary is a context wipe. br epics, tasks, bugs, verify.json step checksums, and cape state flags are the only persistence. V2 leans harder on this: every skill action writes to br or verify.json before stopping, so resumption is always from a known state, never from conversation memory.

V2 br state write discipline

Every skill that touches state must write it to br before handing off to the user. No state lives only in conversation context. 

# Before checkpoint/stop:
cape br design <task-id> "Outcome"
cape br close <task-id>
cape state clear workflowActive

# Resumption (any session):
br list --status in_progress
br ready --parent <epic-id>
cat .beads/<epic>/verify.json

Resumable step checksums

verify.json records HEAD SHA per completed step. On resume, execute-plan skips steps whose SHA matches HEAD — idempotent re-entry after /clear or failure. 

// .beads/<epic-id>/verify.json
{
  "step-1": "a3f8c12",
  "step-2": "b91e4d7"
}

# Gate check before each step:
if sha matches HEAD:
  skip → "Step N already passed"

CI and external integration

Hook → CI bridge

PostToolUse Bash (git push) can trigger a cape CI check that polls the remote status and surfaces failures back as additionalContext. This closes the loop without the user switching context. 

# hooks.json matcher:
{
  "event": "PostToolUse",
  "matcher": "Bash",
  "command": "cape hook ci-push"
}

# cape hook ci-push:
gh run watch --exit-status
echo "CI: PASS" >&2 || \
  cape br create --type bug "CI failed"

br as issue sink for CI

CI failures auto-create br bugs via a webhook or post-push hook. debug-issue then closes the loop: bug already exists, root cause investigation starts immediately. 

# CI webhook → br bug:
cape br create "CI: test_auth fails" \
  --type bug \
  --priority high \
  --description "$(cat ci-failure.log)"

# Model picks up on next session:
br list --type bug --status open
# → routes to debug-issue automatically

cape validate in CI

Run cape validate skills and cape validate agents in CI to catch malformed skill frontmatter before it ships. Skills are configuration — they should be linted like code. 

# .github/workflows/cape.yml:
- run: cape validate
- run: pnpm check

# Catches: missing required fields,
# unknown agent names in dispatch
# tables, broken Skill tool refs.

Conform as pre-push gate

PostToolUse Bash (git push) can run conform branch and block push if violations exist. Stricter than pre-commit; covers multi-commit branches. 

# PreToolUse Bash git push:
cape conform branch --json \
  | jq '.violations | length'

# exit 1 → deny push
# exit 0 → allow push
# Violations appear as hook message

Power-user extension model

Custom hook matchers

hooks.json accepts user-defined matchers. Any tool + pattern combination can invoke a cape CLI command. This is how conform, TDD state, and the CI bridge all work — no core changes needed. 

// hooks/hooks.json
{
  "hooks": [{
    "event": "PreToolUse",
    "matcher": "Bash",
    "command": "cape hook pre-bash"
  }, {
    "event": "PostToolUse",
    "matcher": "Edit",
    "command": "cape hook post-edit"
  }]
}

Skill composition via Skill tool

Skills call other skills via the Skill tool — not by embedding their instructions. This means skill authors can compose workflows without forking core skills. execute-plan → expand-task → TDD → refactor is pure composition; each link is swappable. 

# Swap TDD implementation:
# replace "cape:test-driven-development"
# with "myorg:tdd-strict"
# in execute-plan SKILL.md only.

# All callers inherit the change
# because they load by name.

AFK mode (autonomous pipeline)

Execute-plan's AFK execution mode skips HITL stops and chains tasks automatically. Controlled by the epic's "Execution mode" field. Power users set AFK for well-understood, low-risk epics; keep HITL for exploratory work. 

# Epic design field:
## Execution mode
AFK

# execute-plan then:
# task → commit → next task
# → commit → ... → finish-epic
# No stops until epic closes.

Custom agent wiring

The 5 core agents are dispatched by name. Skills reference agent names in their SKILL.md, not inline instructions. Adding a project-specific agent (e.g., myorg:db-schema-checker) means adding one agent file and one dispatch line in the relevant skill. 

# In expand-task SKILL.md:
# dispatch cape:codebase-investigator
# OR myorg:db-schema-checker
# when task touches DB layer.

# The agent protocol is:
# pass context → expect verdict
# No skill core changes needed.

V2 concrete changes

1 PreToolUse Edit/Write DENY when tdd-state ∈ {null, writing-test}. Moves TDD from advisory PostToolUse warning to blocking PreToolUse deny. Eliminates the 80% of TDD violations that happen when the model is "in flow." hooks/hooks.json · cli/src/services/hook.ts
2 PreToolUse Bash (git commit) DENY when conform staged returns violations. Conform runs 7× manually today; V2 makes it automatic and blocking. Zero-config for users — it just fires. hooks/hooks.json · cli/src/services/hook.ts
3 Cut bug-tracer, notebox-researcher; merge modes into codebase-investigator. 2 dispatches each in 10 days — not worth the agent overhead. codebase-investigator gains --mode=trace-bug and --mode=audit-tests invocation patterns. 8 agents → 5. agents/ directory
4 br write discipline: every skill writes state before stopping. With 37 /clear events, any state held only in conversation context is silently lost. Outcome, next task, cape state flags — all written to br before the checkpoint stop. skills/execute-plan/SKILL.md · skills/finish-epic/SKILL.md
5 Stop hook: if workflowActive, remind user to /clear. Structural checkpoint discipline is non-negotiable but the model forgets. A Stop hook fires after every model turn — the reminder is guaranteed, not probabilistic. hooks/hooks.json · cli/src/services/hook.ts
