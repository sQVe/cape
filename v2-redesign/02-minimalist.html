Skills — Decision table

Skill Uses
(10d)		 Verdict Rationale
commit 29 Keep Most-used skill by far. Sole path to clean conventional commits.
review 12 Keep Strong signal, used equally in pre-commit and PR contexts.
pr 12 Keep Terminal ship step — irreplaceable, no overlap with review.
test-driven-development 11 Keep RED-GREEN-REFACTOR cycle is a genuine discipline enforcer, used in every BUILD chain.
brainstorm 9 Keep Anchors the PLAN chain; challenge always fires inside it (see challenge verdict).
execute-plan 7 Keep Core BUILD driver; dispatches expand-task internally.
conform 7 Keep Post-commit convention check fills a distinct gap review doesn't cover (rules files).
challenge 7 Merge → brainstorm Data shows challenge is always called inside planning, never standalone. Fold adversarial questioning into brainstorm's closing phase. Eliminates a separate invocation with no loss.
write-plan 6 Keep Formalises brainstorm output into br epics — distinct artifact, needed to enter BUILD.
finish-epic 4 Keep Low-frequency-but-necessary: every completed epic needs closure. Not dead — low frequency matches low epic completion rate.
fix-bug 3 Keep Bugs happen infrequently but need a reliable end-to-end path. Keeping; 3 uses in 10 days is a healthy rate for bugs.
beads 3 Keep br integration is foundational to the whole planning/tracking system. 3 direct invocations undercount — it's used implicitly everywhere.
expand-task 2 Merge → execute-plan Internal sub-skill with no user-facing value. 2 invocations were likely accidental direct calls. Inline the expansion logic into execute-plan.
task-refinement 1 Cut 1 use in 10 days. Stress-testing edge cases before implementation is good practice, but nobody's reaching for this skill — it's not part of any established chain. Dead in practice.
refactor 1 Cut 1 use. Behavior-preserving refactoring is real work, but TDD already enforces the safe-refactor cycle. The distinct skill adds surface without adding behaviour.
debug-issue 1 Merge → fix-bug 1 standalone use. Investigation and fixing are the same session in practice. Fold investigation mode into fix-bug (--investigate flag or opening phase).
analyze-tests 0 Cut Zero uses. Test quality concerns are handled implicitly by TDD + review. Dead, not low-frequency-but-valuable.
branch 0 Cut Zero uses. Git branch creation doesn't need a skill — it's a one-liner. Cognitive overhead without value.
design-an-interface 0 Cut Zero uses. Interface design is either handled in brainstorm or is too domain-specific to generalise. Dead.
explain 0 Cut Zero uses. Code explanation is a base Claude capability, not a workflow skill. Wrapper with no added value.
find-test-gaps 0 Cut Zero uses. Gap analysis is valuable in theory, but TDD prevents gaps from accumulating in the first place. Redundant given TDD adoption.

Workflow chains — unchanged structure, fewer steps

Plan brainstorm → challenge → write-plan challenge folded into brainstorm closing phase
Build execute-plan → expand-task → tdd → commit expand-task inlined, invisible to user
Ship finish-epic → pr + conform
Bug fix-bug debug-issue is now its opening investigation phase

Cuts defended — the hard cases

Skill Why "never used" = dead here
analyze-tests / find-test-gaps TDD at 11 invocations means gaps are prevented, not discovered after the fact. These skills assume a workflow that doesn't exist in practice.
design-an-interface Interface design happens inside brainstorm. A separate skill implies a separate session that never materialised in 10 days of real work.
explain Pure base model capability. Wrapping it as a skill adds discovery friction without any specialisation. The CLI shouldn't compete with just asking Claude.
branch `git checkout -b` is faster to type than invoking a skill. Skills justify themselves through complex orchestration, not wrapping one-liners.
